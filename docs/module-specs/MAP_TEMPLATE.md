# MAP v1 Template

Use this structure for every module spec in `docs/module-specs/`.

## 1. Purpose

What this module does and what it explicitly does not do.

## 2. Phase

Which Part 3 phase owns this module.

## 3. Ownership

The primary owner boundary: API, domain, service, frontend, docs, or deployment.

## 4. Inputs And Outputs

Typed inputs and outputs at the module boundary.

## 5. Public Interfaces

Functions, classes, routes, components, or commands other modules may call.

## 6. Dependencies

Internal and external dependencies. Note forbidden dependencies.

## 7. Error Behavior

Expected failures and how the module reports them.

## 8. Tests Required

Unit, service, API, frontend, or manual tests required for the phase exit check.

## 9. Exit Criteria

Observable conditions that prove the module is done for its phase.

## 10. Files Likely Touched

Expected files. Keep the list small and phase-scoped.
