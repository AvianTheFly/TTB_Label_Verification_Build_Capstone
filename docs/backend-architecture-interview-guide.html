Executive Summary

The backend is designed as a stateless FastAPI service with a modular, contract-driven architecture. Each layer has a clear responsibility, which makes the system easier to test, extend, and change without accidentally breaking existing behavior.

Python 3.12 FastAPI Pydantic v2 Pure domain logic Provider boundary No database
Short interview phrasing: I designed the backend with modular components, clear public contracts, and explicit ownership boundaries so new features can be added with less risk of breaking existing functionality.

Architecture

The backend is organized around separation of concerns. API modules handle HTTP details, service modules orchestrate workflows, and domain modules own the business rules. 

backend/app/
  api/        FastAPI route handlers and dependency wiring
  core/       settings, API errors, and exception handlers
  domain/     Pydantic models, normalization, comparison rules
  services/   image preprocessing, vision adapters, verification workflows
  tests/      regression tests for contracts and edge cases

API Layer

Converts HTTP requests into typed inputs, validates uploads, maps exceptions to safe API errors, and returns response models.

api/verify.py, api/batch.py, api/compare.py

Domain Layer

Contains pure business logic: canonical fields, matching strategies, verdict rules, and Pydantic models.

domain/models.py, domain/comparison.py

Service Layer

Coordinates image preprocessing, vision extraction, comparison, timing, and batch concurrency.

use_cases/verification.py, use_cases/batch_verification.py

Provider Boundary

Keeps model-provider code behind a VisionService protocol so tests and local demos do not depend on real external calls.

services/vision.py, services/fake_vision.py

Single-Label Request Workflow

Client submits multipart form data. The request includes an image and application_data JSON containing the seven canonical fields.
The API route parses and validates inputs. api/verify.py checks JSON shape, required fields, upload presence, and upload size.
The image is preprocessed. services/image_preprocess.py validates type, confirms the file is a real image, applies EXIF orientation, downscales, and re-encodes to JPEG.
The vision service extracts structured fields. The provider returns exactly the seven canonical fields. Unknown values are null, not guessed.
The comparison engine evaluates each field. domain/comparison.py applies fuzzy, numeric, unit, synonym, and exact matching strategies.
The service returns a complete verification result. The response includes per-field results, expected/found values, overall verdict, and latency_ms.

Batch Workflow

Batch verification reuses the single-label orchestration instead of creating a separate comparison path. This protects behavior consistency between /verify and /verify/batch.

  • Items are paired by multipart order: image 0 with application data 0.
  • Each item is validated independently.
  • One bad item returns an item-level error and does not fail the whole batch.
  • Concurrency is bounded with an async semaphore to protect provider stability.
  • The summary counts approved results, needs-review results, and item errors.

Comparison Engine

The comparison engine is pure domain logic. It does not import FastAPI, read files, call providers, or depend on network access.

Field Strategy Reason
brand_name Fuzzy match Handles capitalization and punctuation differences.
class_type Fuzzy match Allows small OCR/model differences without approving unrelated classes.
abv Numeric parse Compares values like 45% and 90 Proof.
net_contents Unit normalization Compares equivalent units like 750 mL and 0.75 L.
producer Fuzzy match Handles address punctuation and casing differences.
country_of_origin Synonym normalization Maps examples like USA to United States.
government_warning Exact after whitespace collapse Required by the source requirements; case and punctuation must match.
Important interview point: The government warning intentionally does not use fuzzy matching. A near match still fails because legal warning text exactness is mandatory.

Design Decisions

Why no database?

The MVP is stateless. Each request includes everything needed to verify a label, so there is no persistence risk and no request history dependency.

Why Pydantic models?

Models make API boundaries explicit, reject unexpected fields, and keep backend and frontend contracts aligned.

Why a vision interface?

Provider code can change without rewriting the API or comparison engine. Tests can use fakes and never require real API keys.

Why preprocess images?

Preprocessing controls upload size, validates image content, standardizes format, and reduces provider latency and cost risk.

Why typed error envelopes?

Public errors stay safe for users and the frontend can branch on stable error codes instead of parsing exception text.

Why bounded batch concurrency?

It improves throughput without creating unbounded provider calls, rate-limit risk, or memory pressure.

Testing Strategy

The tests focus on contracts and high-risk edge cases, especially places where AI output could be wrong or inconsistent.

  • Comparison tests cover normalization, ABV, units, country synonyms, exact warning text, and verdict rules.
  • Vision tests use fake or mocked services, not real provider calls.
  • Endpoint tests verify response shape, error envelopes, upload validation, and latency fields.
  • Batch tests verify bounded concurrency behavior, summary counts, and item-level error isolation.
cd backend
uv run --extra dev pytest

Tradeoffs And Limitations

  • The app verifies text, not visual styling. It documents government-warning styling as a limitation unless a future phase adds style evidence.
  • The real vision provider path depends on external model availability, so tests rely on deterministic fake services.
  • Batch request pairing is order-based. This is acceptable for the MVP because the frontend constructs the batch request.
  • The /compare endpoint supports reviewer-edited extracted data. It extends the workflow while keeping comparison logic backend-owned.

Interview Q&A

How would you describe the backend architecture?

It is a modular FastAPI backend with clear boundaries between HTTP handling, business logic, workflow orchestration, and provider integration. The domain layer is pure and testable, while external services are isolated behind interfaces.

What was your most important design decision?

Keeping comparison logic pure and separate from FastAPI and the vision provider. That makes the core business rules easy to test and protects them from changes in upload handling or model-provider behavior.

How did you prevent AI/provider output from reshaping the app?

The provider must return structured JSON mapped into ExtractedLabel. Unknown fields become null, extra fields are rejected, and the comparison engine owns the final decision.

Why is the government warning handled differently?

The requirement says warning text must be exact and case-sensitive after whitespace collapse. Fuzzy matching would risk approving legally incorrect text.

How does batch processing avoid one failure breaking everything?

Each batch item is converted into a BatchVerificationInput. Invalid items carry their own error, valid items run through the normal single-label verification flow, and the response summarizes all outcomes.

What would you improve next?

I would continue hardening accessibility and latency, add deployed end-to-end verification, and consider extracting shared request-parsing helpers from the API modules if the route layer grows.
